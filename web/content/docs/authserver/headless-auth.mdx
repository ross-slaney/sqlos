# Headless Auth

**What this guide covers**: When to use headless mode, how the handoff works, how custom signup fields stay app-owned, and how the SqlOS example demonstrates the pattern end to end.

---

Headless mode keeps the authorization server in SqlOS while moving the authorize UI into your application.

Use it when:

- the hosted AuthPage is too generic for your product
- the OAuth popup should match your real app layout
- you want experiments, conversion copy, and app-owned signup fields
- you still want SqlOS to own `/authorize`, `/token`, PKCE, sessions, refresh, OIDC, SAML, and code issuance

If you are still deciding between hosted and headless, read [Hosted vs Headless Auth](/docs/authserver/hosted-vs-headless).

## Product boundary

Headless mode is not “replace SqlOS auth with your own auth.”

It is:

- **SqlOS owns the protocol**
- **your app owns the presentation**

That is the key difference from the usual “nice login page plus custom bridge code” pattern.

Your app can own:

- HTML and layout
- experiments and copy
- design system integration
- app-specific signup fields

SqlOS still owns:

- authorization request creation
- redirect URI validation
- PKCE validation
- OIDC and SAML callbacks
- auth code issuance
- token exchange
- refresh and logout

## Configuration

```csharp
builder.Services.AddSqlOS<AppDbContext>(options =>
{
    options.UseAuthServer(auth =>
    {
        auth.BasePath = "/oauth";
        auth.Issuer = "https://app.example.com/oauth";

        auth.UseHeadlessAuthPage(headless =>
        {
            headless.BuildUiUrl = ctx =>
                $"https://app.example.com/authorize?request={ctx.RequestId}&view={ctx.View}";

            headless.OnHeadlessSignupAsync = async (ctx, cancellationToken) =>
            {
                var referralSource = ctx.CustomFields["referralSource"]?.GetValue<string>();

                if (string.IsNullOrWhiteSpace(referralSource))
                {
                    throw new SqlOSHeadlessValidationException(
                        "Tell us how you heard about the product.",
                        new Dictionary<string, string>
                        {
                            ["referralSource"] = "Select a referral source to continue."
                        });
                }

                // Persist app-owned profile data here.
                await Task.CompletedTask;
            };
        });
    });
});
```

## Example project

The SqlOS example app is now the canonical headless reference. It demonstrates:

- a full custom authorize shell at `/auth/authorize`
- a real PKCE browser flow through SqlOS `/authorize`
- a custom signup field named `referralSource`
- persistence of that field into example-owned data during `OnHeadlessSignupAsync`
- a signed-in app card that reads the stored profile back through the normal API

### Backend

The example API enables headless mode when `SqlOS:AuthMode` is `Headless`.

```json
{
  "SqlOS": {
    "AuthMode": "Headless",
    "HeadlessFrontendUrl": "http://localhost:3000"
  }
}
```

Or:

```bash
SqlOS__AuthMode=Headless dotnet run --project examples/SqlOS.Example.Api
```

The example’s `OnHeadlessSignupAsync(...)` does real application work:

- validates `referralSource`
- writes an example-owned profile row keyed to the SqlOS user
- keeps the flow fully automatic with no dashboard setup steps

### Frontend

The Next.js example ships a real app-owned authorize experience at `/auth/authorize`.

Key files:

- `app/auth/authorize/page.tsx` — app-owned authorize route
- `components/sqlos-headless-auth-panel.tsx` — custom shell, login, signup, provider, and organization-selection UI
- `components/example-profile-panel.tsx` — signed-in proof that app-owned signup data was persisted
- `lib/sqlos-headless.ts` — typed client for the six existing headless endpoints

When no `request` query parameter is present, the route still renders the full custom shell and starts a real PKCE flow from that UI.

### Try it

1. Start the API in headless mode.
2. Start the example frontend.
3. Visit `http://localhost:3000`.
4. Click **Headless authorize UI** or **Start headless signup**.
5. Complete signup and choose a **Referral source**.
6. SqlOS issues the code and tokens normally.
7. After sign-in, visit `/app` and inspect the **Headless custom field demo** card.

That card is the important part: it proves the signup field belongs to the app while the OAuth flow still belongs to SqlOS.

## Flow

1. OAuth client calls `GET /oauth/authorize?...`
2. SqlOS validates the request and creates an authorization request record
3. SqlOS redirects to the UI URL from `BuildUiUrl(...)`
4. Your UI calls the SqlOS headless endpoints to continue the flow
5. SqlOS authenticates the user and issues the final authorization code
6. The OAuth client exchanges the code at `POST /oauth/token`

## Headless endpoint surface

These endpoints are unchanged in headless mode:

- `GET /oauth/headless/requests/{requestId}`
- `POST /oauth/headless/identify`
- `POST /oauth/headless/password/login`
- `POST /oauth/headless/signup`
- `POST /oauth/headless/organization/select`
- `POST /oauth/headless/provider/start`

The UI treats those endpoints as the state machine for the current authorization request.

## App-owned signup data

SqlOS keeps its core auth tables narrow. App-specific fields belong in `customFields`.

Typical examples:

- referral source
- campaign variant
- company profile data
- invitation code

In the example app, `referralSource` is:

1. captured by the custom UI
2. sent through `customFields`
3. validated in `OnHeadlessSignupAsync`
4. persisted into an example-owned profile table
5. surfaced after sign-in through the example API

That is the pattern to copy in a real product.

## Emcy and MCP

Headless mode does **not** change the OAuth wire flow for Emcy or any other modern OAuth client.

You still expose:

- `/.well-known/oauth-authorization-server`
- `/authorize`
- `/token`
- `/.well-known/jwks.json`

The only thing that changes is what the user sees in the popup:

- hosted mode: SqlOS AuthPage
- headless mode: your app-native authorize UI

That is why headless mode is a good fit for product teams that want a polished Emcy/MCP popup without rebuilding OAuth.
