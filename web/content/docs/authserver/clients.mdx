**What this guide covers**: Client applications, startup seeding, client ID, audience, redirect URIs.

**Dashboard path**: Auth Server → Clients  
**URL**: `http://localhost:5062/sqlos/admin/auth/clients`

---

A client represents an application that authenticates users through SqlOS. Each client has a unique client ID and explicit redirect URIs for browser PKCE flows.

## Fields

| Field             | Required | Description                             |
| ----------------- | -------- | --------------------------------------- |
| **Client ID**     | Yes      | Unique identifier (e.g., `example-web`) |
| **Name**          | Yes      | Display name                            |
| **Audience**      | No       | Token audience claim                    |
| **Redirect URIs** | Yes for browser clients | Allowed callback URIs for OAuth flows   |

## Recommended setup

Seed first-party clients in startup so the app boots into a working hosted auth flow without dashboard setup:

```csharp
builder.Services.AddSqlOS<AppDbContext>(options =>
{
    options.UseAuthServer(auth =>
    {
        auth.SeedBrowserClient(
            "my-web-app",
            "My Web Application",
            "http://localhost:3010/auth/callback",
            "https://app.example.com/auth/callback");
    });
});
```

Dashboard-created clients are still supported. If a client is marked as startup managed in the dashboard, startup code is authoritative and those values are reapplied on restart.
