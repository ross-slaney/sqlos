Application-level authorization checks are essential, but they're not enough. If a bug bypasses your permission check, or a developer forgets to add one, unauthorized data can leak. The solution? Enforce authorization at the database level using row-level security (RLS).

In this guide, we'll implement RLS using SQL Server Table-Valued Functions (TVFs) integrated with Entity Framework Core. This approach gives you defense in depth: even if application code fails, the database won't return unauthorized rows.

## Why Database-Level Security?

Consider this common pattern:

```csharp
// Application-level filtering
var documents = await dbContext.Documents
    .Where(d => d.ProjectId == userProjectId)
    .ToListAsync();
```

Problems with this approach:

1. **Easy to forget**: Every query needs the filter
2. **Easy to bypass**: A bug or oversight exposes data
3. **Hard to audit**: Security logic scattered across codebase
4. **Complex joins**: Filtering through multiple relationships is error-prone

Database-level RLS solves all of these by making unauthorized rows invisible to queries.

## Table-Valued Functions: The Foundation

A Table-Valued Function (TVF) is a SQL Server function that returns a table. We'll use TVFs to return the set of resource IDs a subject can access:

```sql
CREATE FUNCTION dbo.GetAccessibleResources(
    @SubjectId NVARCHAR(100),
    @PermissionKey NVARCHAR(100)
)
RETURNS TABLE
AS
RETURN (
    SELECT DISTINCT r.Id AS ResourceId
    FROM Resources r
    INNER JOIN Grants g ON g.ResourceId = r.Id 
        OR g.ResourceId IN (
            SELECT ancestor.Id 
            FROM Resources ancestor
            WHERE r.Path LIKE ancestor.Path + '%'
        )
    INNER JOIN RolePermissions rp ON rp.RoleId = g.RoleId
    INNER JOIN Permissions p ON p.Id = rp.PermissionId
    WHERE g.SubjectId = @SubjectId
        AND p.Key = @PermissionKey
        AND (g.EffectiveFrom IS NULL OR g.EffectiveFrom <= GETUTCDATE())
        AND (g.EffectiveTo IS NULL OR g.EffectiveTo >= GETUTCDATE())
);
```

This function:
1. Takes a subject ID and permission key
2. Finds all grants for that subject
3. Includes inherited permissions (via the Path column)
4. Returns all accessible resource IDs

## Integrating with EF Core

### Step 1: Register the TVF

First, tell EF Core about the TVF. The key insight is using `FromExpression` to map a C# method to the SQL function:

```csharp
public class AppDbContext : DbContext, ISqlzibarDbContext
{
    public DbSet<Document> Documents => Set<Document>();
    
    public IQueryable<SqlzibarAccessibleResource> IsResourceAccessible(
        string resourceId, string subjectIds, string permissionId)
        => FromExpression(() => IsResourceAccessible(resourceId, subjectIds, permissionId));
    
    protected override void OnModelCreating(ModelBuilder modelBuilder)
    {
        base.OnModelCreating(modelBuilder);
        modelBuilder.ApplySqlzibarModel(GetType());
    }
}
```

### Step 2: Use CROSS APPLY in Queries

The TVF is designed to be used with `CROSS APPLY` (or `JOIN` in LINQ). This allows the database to filter rows as part of the query plan:

```csharp
var documents = await dbContext.Documents
    .Join(
        dbContext.IsResourceAccessible(null!, subjectIds, permissionId),
        doc => doc.ResourceId,
        acc => acc.ResourceId,
        (doc, acc) => doc)
    .ToListAsync();
```

The generated SQL uses CROSS APPLY for efficient filtering:

```sql
SELECT d.*
FROM Documents d
CROSS APPLY dbo.fn_IsResourceAccessible(d.ResourceId, @p0, @p1) acc
```

### Step 3: Use the Specification Executor

For real applications, Sqlzibar provides a higher-level API that handles pagination, sorting, and authorization in one call:

```csharp
var spec = PagedSpec.For<Document>(d => d.Id)
    .RequirePermission("DOCUMENT_VIEW")
    .SortByString("name", d => d.Name, isDefault: true)
    .Search(search, d => d.Name, d => d.Description)
    .Build(pageSize, cursor, sortBy, sortDir);

var result = await executor.ExecuteAsync(
    context.Documents, spec, subjectId,
    d => new { d.Id, d.Name, d.CreatedAt });
```

This generates optimized SQL with the TVF join, pagination, and sorting all in one query.

## Handling Hierarchical Resources

The real power comes from hierarchical filtering. When a user has access to a parent resource, they automatically get access to all children.

### Materialized Paths

Store the full ancestry path in each resource:

```sql
-- Resources table
Id          | Name           | ParentId    | Path
------------|----------------|-------------|------------------
org_1       | Acme Corp      | NULL        | /org_1/
team_eng    | Engineering    | org_1       | /org_1/team_eng/
proj_api    | API Project    | team_eng    | /org_1/team_eng/proj_api/
doc_123     | Design Doc     | proj_api    | /org_1/team_eng/proj_api/doc_123/
```

### Path-Based Filtering

The TVF uses LIKE queries on paths for efficient ancestor matching:

```sql
-- Find all ancestors of a resource
SELECT Id FROM Resources 
WHERE @ResourcePath LIKE Path + '%'

-- Find all descendants of a resource
SELECT Id FROM Resources
WHERE Path LIKE @ResourcePath + '%'
```

With a proper index on the Path column, these queries are fast even with millions of rows.

## Performance Optimization

### Index Strategy

Create these indexes for optimal TVF performance:

```sql
-- Fast grant lookups
CREATE INDEX IX_Grants_Subject_Resource 
ON Grants(SubjectId, ResourceId) 
INCLUDE (RoleId, EffectiveFrom, EffectiveTo);

-- Fast path queries
CREATE INDEX IX_Resources_Path 
ON Resources(Path) 
INCLUDE (Id, ParentId);

-- Fast permission lookups
CREATE INDEX IX_RolePermissions_Role 
ON RolePermissions(RoleId) 
INCLUDE (PermissionId);
```

### Query Plan Analysis

Always check the execution plan for TVF queries:

```sql
SET STATISTICS IO ON;
SET STATISTICS TIME ON;

SELECT d.*
FROM Documents d
WHERE d.ResourceId IN (
    SELECT ResourceId 
    FROM dbo.GetAccessibleResources('user_123', 'documents:read')
);
```

Look for:
- Index seeks (good) vs. scans (bad)
- Estimated vs. actual row counts
- Hash matches vs. nested loops

### Caching Considerations

TVF results can be cached at multiple levels:

1. **SQL Server query cache**: Parameterized queries reuse plans
2. **Application cache**: Cache permission results for hot paths
3. **Materialized views**: For read-heavy workloads, consider indexed views

```csharp
// Application-level caching
var cacheKey = $"accessible:{subjectId}:{permissionKey}";
var accessibleIds = await cache.GetOrCreateAsync(cacheKey, async entry =>
{
    entry.AbsoluteExpirationRelativeToNow = TimeSpan.FromMinutes(5);
    return await dbContext.GetAccessibleResources(subjectId, permissionKey)
        .Select(a => a.ResourceId)
        .ToListAsync();
});
```

## Advanced Patterns

### Multi-Permission Queries

Sometimes you need to check multiple permissions:

```csharp
public static IQueryable<T> WhereSubjectHasAnyPermission<T>(
    this IQueryable<T> query,
    AppDbContext context,
    string subjectId,
    params string[] permissionKeys) where T : IHasResourceId
{
    var accessibleIds = permissionKeys
        .SelectMany(pk => context.GetAccessibleResources(subjectId, pk))
        .Select(a => a.ResourceId)
        .Distinct();
        
    return query.Where(e => accessibleIds.Contains(e.ResourceId));
}
```

### Temporal Permissions

Handle time-bound access with effective dates:

```sql
-- In the TVF
WHERE (g.EffectiveFrom IS NULL OR g.EffectiveFrom <= GETUTCDATE())
  AND (g.EffectiveTo IS NULL OR g.EffectiveTo >= GETUTCDATE())
```

This enables:
- Scheduled access grants (starts next Monday)
- Temporary access (expires in 24 hours)
- Access revocation without deleting grants

### Group Membership

Expand subject IDs to include group memberships:

```sql
CREATE FUNCTION dbo.GetEffectiveSubjectIds(@SubjectId NVARCHAR(100))
RETURNS TABLE
AS
RETURN (
    SELECT @SubjectId AS SubjectId
    UNION
    SELECT gm.GroupSubjectId
    FROM GroupMemberships gm
    WHERE gm.MemberSubjectId = @SubjectId
);
```

Then use it in the main TVF:

```sql
WHERE g.SubjectId IN (SELECT SubjectId FROM dbo.GetEffectiveSubjectIds(@SubjectId))
```

## Security Considerations

### SQL Injection Prevention

Always use parameterized queries. EF Core handles this automatically when using DbFunctions:

```csharp
// Safe: EF Core parameterizes the values
var results = context.GetAccessibleResources(subjectId, permissionKey);

// Dangerous: Never do this
var sql = $"SELECT * FROM dbo.GetAccessibleResources('{subjectId}', '{permissionKey}')";
```

### Defense in Depth

TVF-based RLS is one layer. Combine with:

1. **Application checks**: Validate before expensive operations
2. **API authorization**: Protect endpoints with policies
3. **Audit logging**: Track all access attempts
4. **Encryption**: Protect sensitive data at rest

### Testing

Test your TVFs thoroughly:

```csharp
[Fact]
public async Task User_CannotAccess_UnauthorizedResources()
{
    // Arrange
    var unauthorizedDocId = "doc_other_team";
    
    // Act
    var accessible = await dbContext.GetAccessibleResources(
        "user_alice", 
        "documents:read"
    ).ToListAsync();
    
    // Assert
    Assert.DoesNotContain(accessible, a => a.ResourceId == unauthorizedDocId);
}
```

## Conclusion

Row-level security with TVFs provides a robust foundation for authorization:

- **Defense in depth**: Database enforces security even if application fails
- **Performance**: Filtering happens at the data source
- **Consistency**: One place for authorization logic
- **Auditability**: Clear, testable security rules

Combined with hierarchical RBAC, you get a complete authorization system that's both powerful and secure. The database becomes your last line of defense, ensuring that unauthorized data never leaves the server.

Start with simple TVFs, measure performance, and optimize as needed. The investment in database-level security pays dividends in reduced vulnerabilities and simplified application code.
